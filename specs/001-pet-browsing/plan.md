# Implementation Plan: Pet Browsing

**Branch**: `001-pet-browsing` | **Date**: 2026-05-05 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/001-pet-browsing/spec.md`

## Summary

Enable users to browse available pets in the PetStore ecommerce application through a responsive catalog interface supporting pagination, category filtering, text search (exact match), and detailed product views. This MVP focuses on discovery only—no shopping cart or checkout. Implements React frontend with Material-UI + Tailwind CSS connected to Java Spring Boot REST API backed by PostgreSQL. Follows API-First, containerized, and component-based design principles per PetStore Constitution.

## Technical Context

**Language/Version**: 
- Backend: Java 17+ with Spring Boot 3.2+
- Frontend: React 18+ with TypeScript
- Runtime: Node.js 18+

**Primary Dependencies**: 
- Backend: Spring Boot Web, Spring Data JPA, Spring Security (for future authentication), Lombok
- Frontend: React Router, Axios, Material-UI (MUI) v5, Tailwind CSS, React Query
- Database: PostgreSQL 14+
- API Documentation: OpenAPI 3.0 / Swagger

**Storage**: PostgreSQL 14+ (single source of truth per Constitution II)

**Testing**: 
- Backend: JUnit 5, Mockito, TestContainers (for PostgreSQL integration tests)
- Frontend: Jest, React Testing Library, Cypress (E2E)

**Target Platform**: 
- Web browsers: Chrome, Firefox, Safari, Edge (last 2 versions)
- Deployment: Render platform with containerized services

**Project Type**: Web application (fullstack ecommerce)

**Performance Goals**: 
- Page load: ≤ 3 seconds (SC-001)
- Search response: ≤ 1 second (SC-005)
- Support 10,000 concurrent users (per Constitution assumptions)

**Constraints**: 
- Exact-match search only (no fuzzy/partial matching)
- Single-category filter (not multi-select)
- Desktop-first responsive design
- No personalization or recommendations in v1

**Scale/Scope**: 
- 20+ pets across 4 categories (MVP)
- 4 user stories (3 P1 MVP + 1 P2 search)
- ~8-12 components (React)
- ~5-7 API endpoints (backend)
- ~40-50 database migrations and models

**Objective**: Resolve all unknowns and research best practices for pet browsing implementation.

## Research Tasks

### R1: React + Material-UI + Tailwind CSS Best Practices
**Topic**: Frontend component architecture using MUI + Tailwind integration  
**Unknowns Resolved**:
- How to effectively combine Material-UI (pre-built components) with Tailwind CSS (utility-first styling)?
- Best practices for theming and customization?
- Performance considerations for both libraries together?

**Findings**:
- Use MUI components for complex UI (cards, dialogs, menus) and Tailwind for custom utility styling
- MUI v5 supports custom theme with Tailwind tokens via `sx` prop and `styled()` API
- Keep component logic in React; use MUI's sx prop for MUI components, Tailwind classes for custom elements
- Import only used MUI components to minimize bundle size
- Tailwind's JIT compilation handles dynamic classes efficiently

**Decision**: Use MUI for complex components (PetCard, FilterPanel); Tailwind classes for layout, spacing, custom styling

---

### R2: Spring Boot REST API for Product Catalog
**Topic**: Designing REST endpoints for pet catalog with pagination, filtering, search  
**Unknowns Resolved**:
- API design patterns for catalog endpoints?
- Query parameter conventions (page, size, category, search)?
- Error handling and response formats?

**Findings**:
- Use `/api/v1/pets` for versioning consistency
- Pagination: `?page=0&size=10` (zero-indexed pages, defaults to size 10)
- Filtering: `?category=cats` (single category per spec)
- Search: `?search=fluffy` (exact name match per clarification)
- Response format: `{data: [...], pagination: {page, size, total}}`
- Error format: `{error: "...", code: "...", timestamp: "..."}`

**Decision**: Implement RESTful endpoints with consistent pagination, filtering, search query patterns per API-First principle

---

### R3: PostgreSQL Schema for Pet Catalog
**Topic**: Database schema for products, categories, images with proper relationships  
**Unknowns Resolved**:
- Table relationships and normalization?
- Indexing strategy for search and filtering?
- Handling inventory/availability?

**Findings**:
- Separate tables: categories, pets (products), pet_images
- Foreign keys: pets.category_id → categories.id, pet_images.pet_id → pets.id
- Indexes on: pets(name, category_id), pets(availability_status)
- Inventory tracked via pet.stock_quantity and pet.availability_status
- Migrations use Flyway for version control and reproducibility

**Decision**: Normalized schema with proper foreign keys, indexes on frequently queried columns, Flyway migrations

---

### R4: Pagination Implementation Patterns
**Topic**: Best practices for frontend and backend pagination  
**Unknowns Resolved**:
- Cursor-based vs. offset-based pagination?
- State management for pagination + filtering + search?
- URL state preservation?

**Findings**:
- Use offset-based pagination for simplicity (page, size parameters)
- Preserve filters, search, page in URL query string for bookmarking/sharing
- Frontend: React Router useSearchParams to parse URL; update on filter/page change
- Backend: Use Spring Data JPA Pageable interface; extract parameters from request

**Decision**: Offset-based pagination with URL-driven state; backend uses Spring Data Pageable

---

### R5: React State Management for Catalog Filters & Search
**Topic**: State management approach for multiple filters, search, pagination state  
**Unknowns Resolved**:
- Local component state vs. Context API vs. external store (Redux)?
- How to sync URL with state?

**Findings**:
- Use React Router's useSearchParams for URL sync (lightweight, no external dependency)
- Lift state to top-level page component (PetCatalog); pass via props or Context if deeply nested
- Context API sufficient for this scope (browsing only); Redux overkill for MVP
- Custom hooks (usePets, useFilter) can encapsulate state logic

**Decision**: Combine URL query params (useSearchParams) + React Context for filter/search state

---

### R6: Exact-Match Search Implementation
**Topic**: Backend search logic for exact pet name matching  
**Unknowns Resolved**:
- SQL query pattern for exact match?
- Case sensitivity handling?
- Performance with large datasets?

**Findings**:
- Use SQL `WHERE name = ? COLLATE NOCASE` for case-insensitive exact match
- Database index on name column ensures O(log n) lookup
- For 1000+ pets, performance still <100ms with proper indexing
- PostgreSQL: Use `ILIKE '%term%'` for case-insensitive; for exact match use `ILIKE 'term'` with word boundaries

**Decision**: Backend returns empty result for partial matches; frontend shows "no results" message per spec

## Phase 1a: Data Model

**File**: `data-model.md` (to be created)

### Entities

**1. Category**
- **Purpose**: Classify pets (Cats, Dogs, Fish, Birds)
- **Attributes**:
  - `id: UUID` (primary key)
  - `name: String` (unique; Cats, Dogs, Fish, Birds)
  - `slug: String` (URL-friendly; cats, dogs, fish, birds)
  - `description: String` (optional)
  - `createdAt: Timestamp`
  - `updatedAt: Timestamp`
- **Relationships**: One-to-Many with Pet

**2. Pet (Product)**
- **Purpose**: Individual pet product for sale
- **Catalog View Attributes** (displayed in list):
  - `id: UUID` (primary key)
  - `name: String` (indexed for search)
  - `category_id: UUID` (foreign key → Category)
  - `price: Decimal` (formatted to 2 decimal places)
  - `description: String` (max 500 chars, shown in catalog)
  - `primary_image_id: UUID` (foreign key → Image; first image in gallery)
  - `availability_status: Enum` (IN_STOCK, OUT_OF_STOCK)
- **Detail View Attributes** (additional):
  - `detailed_description: String` (full product description)
  - `stock_quantity: Integer` (inventory count)
  - `characteristics: JSON` (pet-specific attrs: breed, age, color, size, tank-type, etc.)
  - `createdAt: Timestamp`
  - `updatedAt: Timestamp`
- **Relationships**: 
  - Many-to-One with Category
  - One-to-Many with Image

**3. Image**
- **Purpose**: Product images for display in catalog and detail view
- **Attributes**:
  - `id: UUID` (primary key)
  - `pet_id: UUID` (foreign key → Pet; can be null temporarily)
  - `url: String` (cloud storage URL; e.g., S3, Render hosted)
  - `alt_text: String` (accessibility; "Fluffy the orange cat")
  - `sort_order: Integer` (for image gallery sequence; 1 = primary)
  - `createdAt: Timestamp`
- **Relationships**: Many-to-One with Pet

### Database Schema (Pseudocode)

```sql
CREATE TABLE categories (
  id UUID PRIMARY KEY,
  name VARCHAR(50) NOT NULL UNIQUE,
  slug VARCHAR(50) NOT NULL UNIQUE,
  description TEXT,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE pets (
  id UUID PRIMARY KEY,
  name VARCHAR(255) NOT NULL,
  category_id UUID NOT NULL REFERENCES categories(id) ON DELETE RESTRICT,
  price DECIMAL(10, 2) NOT NULL,
  description VARCHAR(500) NOT NULL,
  detailed_description TEXT,
  primary_image_id UUID,
  availability_status VARCHAR(20) DEFAULT 'IN_STOCK',
  stock_quantity INTEGER DEFAULT 0,
  characteristics JSONB,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE pet_images (
  id UUID PRIMARY KEY,
  pet_id UUID NOT NULL REFERENCES pets(id) ON DELETE CASCADE,
  url TEXT NOT NULL,
  alt_text VARCHAR(255),
  sort_order INTEGER,
  created_at TIMESTAMP DEFAULT NOW()
);

-- Indexes for performance
CREATE INDEX idx_pets_name ON pets(name);
CREATE INDEX idx_pets_category_id ON pets(category_id);
CREATE INDEX idx_pets_availability_status ON pets(availability_status);
CREATE INDEX idx_pet_images_pet_id ON pet_images(pet_id);
```

### Relationships & Constraints

- **Category → Pet**: One category has many pets; deleting category is restricted (prevents orphaned pets)
- **Pet → Image**: One pet has many images; deleting pet cascades deletion to images
- **Primary Image**: Pet.primary_image_id links to one Image; can be null (uses placeholder on frontend)

### GET /api/simeon/pets (List Catalog)

**Purpose**: Retrieve paginated list of pets with optional filtering and search

**Request Parameters**:
```
page: integer (0-based, default 0)
size: integer (items per page, default 10, max 100)
category: string (optional; cats, dogs, fish, birds)
search: string (optional; exact pet name match)
```

**Response Format**:
```json
{
  "data": [
    {
      "id": "uuid",
      "name": "Fluffy",
      "category": "cats",
      "price": 49.99,
      "description": "Orange tabby cat, 2 years old",
      "primaryImage": {
        "id": "uuid",
        "url": "https://...",
        "alt": "Fluffy the orange cat"
      },
      "availabilityStatus": "IN_STOCK"
    }
  ],
  "pagination": {
    "page": 0,
    "size": 10,
    "total": 25,
    "totalPages": 3

}
```

**Error Responses**:
- 400: Invalid page/size parameters
- 500: Server error
